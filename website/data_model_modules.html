The KNOT Data Model - Modules

KNOT-DM can be treated as a set of interlinked modules that describe different aspects of academic research projects within the context of the KNOT project. Each module works across one or more of the three segments that make up the data model (see the KNOT-DM structure) and relates to one of the following primary usages:

  • Research project activity — who, what, where, when, and how.
  • Research project outputs — the digital objects created by research project activities.
  • Research project inputs — the entities used by a research project.
  • Agents — the people or organizations who create and publish research project and outputs.
  • Relationships — between activities, agents, inputs, and outputs.
  • Spatial and temporal information — both that of the projects themselves but also contained within the objects it creates.
  • Themes — related to activities and outputs.

It is recommended to first read the Data Model Structure page before the modules, though some information is repeated across both.

The following legend applies to all graphical representations and textual higlights based on the "Graphical Framework for OWL Ontologies" [1]: a yellow box indicates a class, a blue arrow indicates an object property, a green arrow indicates a data property, and a black arrow indicates a predicate between two entities.

The modules makes use of the ALCIDE analysis platform, by the DH@FBK laboratory, as an example project.

Academic Activity as Provenance

Within KNOT-DM academic research projects are conceptualized as a form of activity that uses and generates entities. This activity is described using elements from PROV-O within the Academic Provenance Segment of the data model, with connections to the other two segments via PROV-O properties prov:wasGeneratedBy (connecting to the Public Data Segment) and prov:wasInformedBy (connecting to the Cultural Heritage Information segment).

KNOT-DM does not attempt to record all possible information about the research project as activity but rather a basic level that is almost always publicly available in a project’s documentation such as where and when the project took place, who was involved, and what ‘things’ were used and or generated.

While PROV-O itself allows for a level of specificity that would enable detailed description of what happened during a research project, the how of the activity, this is outside the scope of KNOT-DM. Instead, further details about the activity, and in particular what types of tasks were undertaken and how they relate to cultural heritage information, are described using CIDOC CRM within the Cultural Heritage Information Segment to create a clearer connection between research project activity and the central theme of the KNOT project. See part 2 on Activity as Cultural Heritage for more details.

The description of academic activity in KNOT-DM is centered around four PROV-O classes — Activity, Entity, Location, and Agent — and should provide answers to who, what, where, and when. It’s recommended to begin by describing the individual representing the activity itself before moving to the other classes in any order.

The Academic Activity

prov:Activity represents the overall research project, and the following basic information about it should be recorded: its title (dct:title) and the dates at which it began and ended (prov:startedAtTime, prov:endedAtTime).

For the title, it is recommended to use the one that is most clearly evident in the project’s public documentation with the understanding that it may differ from how the project is referred to elsewhere. In our example, the project is referred to as ALCIDE (ANALYSIS OF LANGUAGE AND CONTENT IN A DIGITAL ENVIRONMENT) on its official page within the research lab’s website while the website itself, which gives access to some of the project’s outputs and documentation, is titled Alcide with the subtitle “Un viaggio negli scritti di De Gasperi." However the website clearly references the ALCIDE acronym and therefore this should be taken as the title of the project.

For the start and end date, the format is YYYY/MM/DD (represented as xsd:date). As this can sometimes be difficult to ascertain at the least a year should be recorded for the beginning and/or end of the project, with 01/01 for the month and date. At most the correct or approximate beginning and end dates should be recorded (again using 01/01 if either month or date is unknown). In our example, the documentation page published by the research lab clearly states the beginning and end dates however the website does not.

The Entities — What did the activity do?

prov:Entity represents both the outputs of a research project activity, that is the objects it created, and the inputs, or the things it may have used to do this which can include physical or digital objects as well as themes (representing concepts or historical figures and periods). As the KNOT project is focused on digital cultural heritage, the scope of the model is to record information about outputs of research activity as digital objects only. The following information about entities should be recorded: a title (dct:title if they have an official title or rdfs:label for the title in the context of the record), a value (prov:value), and its relationship to the activity o another entity (prov:generated, prov:used, prov:wasInfluencedBy, prov:wasDerivedFrom).

Going back to our example, ALCIDE’s goal was to create an online platform to allow analysis of political discourse, using the writings of Italian politician Alcide de Gasperi. Therefore the primary output of the research project activity is a website, which gives users access to a platform where various automated analytical tools can be used to explore de Gasperi’s writings. In addition, we also know that the project created a corpus of texts, used to power the platform, based on a four-volume published edition of the writings of de Gasperi by the publisher Il Mulino. This corpus is not explicitly linked in the documentation available on the project website, however it is publicly available on the GitHub of the DH@FBK lab and should therefore be included in the description, in line with one of the goals of the KNOT project which is to help bring together the products of academic research that are often dispersed online and separated from each other.

Instances of the corpus and the website are therefore members of the prov:Entity class. In addition, they are also members of classes from DCAT (see the Outputs module for more details on these classes), which is the primary standard used in the Public Data Segment of KNOT-DM. This is because KNOT-DM uses the prov:generated (and its inverse) property to connect these two segments so that entities generated by an activity can also be described as examples of public data. As such, KNOT-DM represents individuals using multiple classes based on the requirements of each segment and the specificities of each project (see the Conceptually Similar Classes section for more details). In this particular case, the corpus is also a dcat:Dataset, being a collection of data available online. The website is also a dcat:Catalog, which is the representation of the research project as a container for its outputs including data(sets) and data services. And finally the part of the website that gives users access to the analytical tools is also a dcat:DataService, which is the representation of interactive access to the data. All three individuals are connected to the prov:Activity via the prov:generated property. Lastly, the corpus was created using the digital version of an edited collection of de Gasperi’s writings, which can therefore also be recorded as a prov:Entity. However it is connected to the activity via the prov:used property .

The title for each prov:Entity should be taken from its corresponding object however when a title isn’t clearly available, such as in the case of the website and data service, a descriptive title can be assigned for use in the context of the record being created. In our example, both the corpus and edition have official titles while the website and data service share the same title as the activity, so we can label their respective individuals “ALCIDE website" and “ALCIDE platform" to make it clear what they are a representation of (we could also use the Alcide - Un viaggio negli scritti di De Gasperi title for the website and platform, though that is quite long and perhaps impractical for later management of the metadata).

The value property should always point to a URI that is an accessible representation of the entity. In this sense prov:value acts in a similar way as owl:sameAs.

See the Relationships module for how the Academic Provenance Segment of KNOT-DM can be used to provide further details on the connections that can be described between entities that are used and entities that are generated within an activity such as for example derivation or influence.

Location — Where did the activity take place?

prov:Location represents the location where the activity took place. PROV-O offers a degree of flexibility in how location is conceptualised, includihng non-geographical places, however in most cases KNOT-DM is interested in the geographical location of the agents during the activity, which is extrapolated from their affiliation.

The following information should be recorded about locations: a string representing its name (rdfs:label), a URI pointing to its equivalent in Geonames to act as authority control (owl:sameAs), and its relationship to the activity (prov:atLocation).

In the case of our ALCIDE example, the two laboratories involved in the project are both based in Trento, therefore it is the location where the activity took place.

Agents — Who contributed to the activity?

prov:Agent represents people or organizations who took part in an activity, with the implication that they had some sort of responsibility whether towards the activity itself, towards an entity, or towards another agent. The following information should be recorded about agents: their name (rdfs:label), a URI pointing to its equivalent in a recognised authority control such as Orcid (owl:sameAs), and their relationship with the activity and/or entities and agents involved in the activity (prov:wasAssociatedWith, prov:actedOnBehalfOf).

Academic research projects can often involve a multitude of agents, in a variety of roles. To this end, KNOT-DM restricts the agents that must be recorded to authors and contributors, the people directly involved in the research, and publishers, the agents who made the results of the research available (most often legal entities such as universities or laboratories). In the case of people if a clear affiliation exists this should also be recorded.

Returning once again to the ALCIDE example, the website details a working group of 6 people who acted as contributors and / or authors as well as their affiliations. As such each of these should be included in the description with the prov:wasAssociatedWith property used to connect them to the activity while the prov:actedOnBehalfOf property can be used to connect them to their relevant affiliation, in this case two laboratories from the Bruno Kessler Foundation and the universities of Trento and Cattolica. Each affiliation can in turn also be described as an instance of prov:Agent, though this is optional and a URI pointing to the entity can be used instead of creating a new individual.

Academic Activity as Cultural Heritage Activity

KNOT-DM allows for the further description of academic research activity within its Cultural Heritage Information Segment, which is intended to provide cultural heritage-specific information about the research project activity and the entities it used and/or generated. This segment uses the CIDOC CRM standard, and in particular two of its extensions: CRMdig and LRMoo (FKA FRBRoo).

The description of research activity as cultural heritage is centered around crmdig:D7_Digital_Machine_Event and crmdig:D1_Digital_Object (along with some of their subclasses), which are taken from CRMdig. These classes provide answers to who and where in a way that is similar to PROV-O while giving more detail about the how of what happened. It’s recommended to begin by describing the activity using PROV-O before moving to CIDOC, with the understanding that sometimes specific information about what happened during a research project may not be available and therefore the use of this particular segment may be limited. On the other hand, one of the reasons for basing this segment on CIDOC CRM is that it opens up the possibility to expand some of the descriptions of entities used and generated into the wider cultural heritage domain though this is outside the scope of this documentation.

Tasks / sub-activities - What happened during the activity?

The D7 class represents sub-activities, or tasks, within a larger prov:Activity. This is done via the prov:wasInformedBy property which connects two instances of prov:Activity and which within KNOT-DM has been extended to include D7 in its range. This connection provides “dependency information" by suggesting “that the informed Activity used an Entity that was generated by the informing Activity" without explicitly detailing the entity [2]. Within KNOT-DM this chain of provenance is used to enable the description of sub-activities (or tasks) such as for example the digitalisation of a physical object into a digital one or the creation of a digital object from other digital objects.

D7 is intended as a generic class for sub-activities with the implication that they were conducted by a human agent, in a specific place, on a device such as a computer, and used a digital object and/or software to create a new digital object. The following basic information about these activities should be recorded: a title (rdfs:label), which digital objects they used and/or created (crmdig:L10_had_input, crmdig:L11_had_output, crmdig:L23_used_software), where they took place (crm:P7_took_place_at), and who they were carried out by (P14_carried_out_by).

Depending on the level of information available additional information can also be recorded using D7 subclasses. In the case of a physical object being digitized, crmdig:D2_Digitization_Process can also record the physical object it used, represented by crm:E18_Physical_Thing and the one it created, represented by crmdig:D9_Data_Object (a subclass of D1). The physical object can be further described with LRMoo by using the Item (frbroo:F5_Item, a subclass of E18) and Manifestation (frbroo:F3_Manifestation) entities of the WEMI model of FRBR, which in turn allows the connection of the manifestation to the entity it is about. Meanwhile crmdig:D10_Software_Execution can be used to describe activities where a computer used software or computing operations on a digital object, such as for example creating data visualizations or a dataset, recording both the source digital object and the software it used if that information is available.

Returning once more to ALCIDE, we know that the project used a digital version of a four-volume edition of de Gasperi's writings to create a corpus before incorporating it into an analysis platform. We can therefore detail each of these steps as a sub-activity that informed the overall prov:Activity individual we described in part 1. The digitization of the editions (an instance of D2) was done by the publisher, il Mulino, while the creation of the corpus (an instance of D10) and the creation of the platform incorporating the corpus (an instance of D7) were both done by the agents already described in part 1. D2 took place in Bologna (home of the publisher) while D10 and D7 took place in Trento, which was also already described. We can add classes from CIDOC to existing instances and create the ones we're missing.

The Objects - What did the activities use and create?

crmdig:D1_Digital_Object represents the digital objects that the sub-activities used and/or created, while its subclasses crmdig:D9_Data_Object refers specifically to digital objects created by a D2 activity and crmdig:D14_Software refers to software used or created by an activity. The following information should be recorded about them: a title (rdfs:label), unless the object is already an instance of a class from another segment with an existing title, and their relationship to activities and other entities, such as digital objects being composed of each other (crm:P106_is_composed_of) or activities using them as inputs or creating them as outputs.

As mentioned abobe, physical objects used in digitization processes can be represented with E18 or, if the necessary information is available, the subclass F5 in order to connect the item to its F3 Manifestation and in turn to an instance of crm:E1_Entity that represents its subject. The following information should be recorded about them: a title (rdfs:label) and a URI that is an accessible representation of the entity (owl:sameAs).

Continuing with ALCIDE, we will assume that the D2 activity of digitizing the editions of de Gasperi’s writings used an instance of F5 which resulted in the creation of an instance of D9, representing the raw digital text. In turn this was used as a source by the D10 activity of creating the corpus resulting in the output of an instance of D1 representing the final corpus. This D1 instance is composed of the D9 instance and can be connected to the previous instance of the corpus created in the Academic Provenance Segment by adding the D1 class to the existing individual. Lastly the corpus was used as input by the D7 activity of creating the ALCIDE platform resulting in the output of another instance of D1 representing the part of the website where users can interact with the data — which again can be connected to the previously created individual and to the corpus by the P106 is composed of property from CIDOC CRM.

Lastly, the F5 instance can be connected to an instance of F3 representing the overall content and form of presentation of the editions and which can in turn be connected to an instance of E1 representing Alcide de Gasperi, whom the manifestation is about. At this point further details about de Gasperi can be provided using the CIDOC model, though this is currently outside the scope of this description, and at a minimum a title, representing the name of the entity, and a URI pointing to its equivalent in a relevant authority control should be provided.

Research Outputs as Public Data

Research outputs within KNOT-DM are conceptualized as entities that can be represented by classes from all three of the segments, with each class adding a certain contextual layer to the entity. This module deals with the outputs’ qualities as public data information, see the activity module for details of how to describe outputs in the context of academic provenance and cultural heritage.

The description of outputs as public data makes use of the majority of the DCAT model to record information such as their technical and conceptual characteristics, their relationship to the research project that created them, the spatio/temporal information they contain or refer to, and how they can be accessed and/or reused. It is connected to the Academic Provenance segment via the prov:wasGeneratedBy property and can be connected to the Cultural Heritage Information Segment via the dct:relation property if need be.

The description of outputs as public data is centered on two DCAT classes — Catalog and Dataset — with another two classes — Distribution and Data Service — being used to describe how to access the data. Each class can have a varying level of mandatory, recommended, and optional information about it based on the DCAT AP guidelines with some KNOT-specific updates.

Together these classes provide answers to who, what, where, when, and how as it relates to data (rather than activity). It’s recommended to begin by describing the individual representing the catalogue before moving to the dataset(s) and then onto the distribution and data service (detailed in the access module).

The Research Outputs Container

*In the rare case where a research project does not have a website, this class can represent any digital item that records information about it and/or may give access to its datasets and/or data service such as an entry in a repository.

dcat:Catalog represents a container for all research outputs and documentation. It must be a real digital object that either contains metadata information about the project and its outputs or hosts the data and services it describes. Most often its real world representation will be the research project’s official website.*

The following information must be recorded about it: a description (dct:description), its title (dct:title), its publisher(s) (dct:publisher), the dataset(s) it includes (dcat:dataset), the knowledge organization systems used to classify its dataset(s) (dcat:themeTaxonomy), and the concepts used to classify the catalogue itself (dcat:theme).

Recommended information that should be recorded, if available, includes: the data service(s) it includes, its homepage, the language used in the metadata of the dataset(s), the license under which the catalogue can be (re)used, its release date and/or last modified date, its creator(s), and the geographical area covered by the catalogue. Optional information that can be recorded, if available, includes: keywords, whether or not the catalogue includes another, whether or not it includes a metadata record related to its registration within a system, and a rights statement.

Most, if not all, of this information should be available from the existing public documentation of the research project, starting with any official website (which as mentioned above would be the catalogue being documented). When something is not clear, efforts should be taken to use the most consistent and logical information available. Dates are formatted as YYYY/MM/DD (xsd:date).

Using our example project, ALCIDE, its public website would be the catalogue we want to record as it hosts the dataset and data service alongside information about the project itself that can be used for the needs of the data model. As will often be the case, some information is missing. In this case the license, rights, and release and modification dates, though the latter can be inferred via additional documentation on the publisher’s website.

The Research Outputs as Datasets

dcat:Dataset represents the digital objects created by a research project, acting similarly to the PROV-O and CRMdig classes detailed in the activity module but focused on their qualities as public data.

Based on the DCAT definitions (see the KNOT-DM structure overview), this class is intended to be flexible enough to cover most types of digital object a project may create regardless of their form. As such, while the word dataset may imply a certain limitation within KNOT-DM it would cover everything from a pdf of a paper to an annotated training set or a piece of software (which in effect is a collection of code, data, and documentation).

Importantly, within the W3C DCAT specification, dcat:Dataset is the superclass of dcat:Catalog. Therefore, in the likely case that the official website of a project is the catalogue, it would also be a dataset that the project created though this does not need to be explicitly stated as it is inferred.

The following information must be recorded about a dataset: a description, its title, its type (dct:type), and the concepts used to classify it.

Recommended information that should be recorded, if available, includes: an available distribution (dcat:distribution), its publisher(s), its creator(s), the spatial and temporal (dct:temporal) information it covers, its license (dct:license), an identifier within the context of the catalogue, and the activity that generated it (prov:wasGeneratedBy).

Optional information that can be recorded, if available, includes: keywords, access right details, specification(s) it conforms to, documentation, updating frequency, whether or not it is a version of or is versioned by another dataset, external entities that reference it (such as papers), the language(s) it includes, external identifiers, and its release date and/or last modified date.

In the case of ALCIDE, we know from the published papers about the project that it created one dataset: a corpus of de Gasperi’s writings. As mentioned in the activity module, this corpus is not directly linked from the project’s official websites but is publicly available from the research laboratory’s GitHub. Between those two sources we have access to most of the information we require, aside from the license.

Accessing Outputs as Public Data

Following from the description of research outputs as Public Data, the other two principal DCAT classes used by KNOT-DM — Distribution and Data Service — cover the concept of the data’s accessibility.

While these two classes share some conceptual similarities in what they describe, they also have some fundamental differences, which are covered in detail in the KNOT-DM structure overview. A simple summary is that a distribution can only ever represent a downloadable form of a dataset, and therefore cannot exist without the dataset, while a data service represents all other forms of access to the dataset, and exists as its own conceptual entity (like a catalogue or dataset).

The Data Service — Enhanced Access to Data

dcat:DataService represents access to a dataset via a collection of operations that entail data processing and/or interaction, best summarised as “offering smarter, more interactive ways to the data.” [3] The most common real world representation would be a search interface, which enables a user to access and potentially interact with the data, or an API, which offers more advanced ways to access and (re)use the data.

One important thing to note is that within KNOT-DM, a data service may also represent another type of output from a project even though it is not a dataset because it is often created as part of a research project in order to allow users enhanced access to the data.

The following information must be recorded about a data service: a title, its location as a URL (dcat:endpointURL), its type and/or the concepts used to classify it, and the dataset it serves (dcat:servesDataset).

The recommended information that should be recorded, if available, is a resource that describes the services available at the endpoint (dcat:endpointDescription), such as for example a documentation page on the website. Optional information that can be recorded, if available, includes: access right details, a description of the data service (which may reuse the description of the catalogue if the service is also part of the website that acts as catalogue), usage license, keywords, and its release and/or last modified date.

Continuing with ALCIDE, our example project, the data service is represented by the part of the official website located at /platform which gives a user access to a suite of analytical tools with which to explore the textual corpus. This service therefore also represents one of the outputs of the project, as described in the activity modules, making it a member of the prov:Entity class in addition. All the information needed for its description is available from the website as well as previously described elements of the project.

Distribution — Downloading the Dataset

dcat:Distribution represents a downloadable form of a dataset and, as mentioned above, therefore cannot exist without it. Furthermore, a data service is not required to make a distribution available. This class is therefore optional.

The main thing to note about using this class is that every downloadable instance of a dataset requires its own instance of dcat:Distribution, as every download may in fact offer different digital representations of a dataset, whether that be parts of it or different encodings.

The following information must be recorded about it: an access URL (dcat:accessURL) and its format (dct:format). Recommended information that should be recorded, if available, includes: a description and a release license. Optional information that can be recorded, if available, includes: a data service that gives access to the distribution, its byte size (using xsd:decimal), its compression format, a page documenting it, a direct download URL, a language used in the distribution, a standard to which it conforms, its title, its release and/or last modified date, a related rights statement, and its status.

In the case of our example, we know that the ALCIDE dataset has three distributions available via the GitHub repository that details the corpus. Each distribution represents a different format under which the corpus is available, with all the necessary information available via the repository. The main features of these distributions is that they do not have a connected data service, because the ALCIDE platform does not give access to them, that they are all compressed using ZIP, and that they are all released under a CC license, which is the only license/rights information about the project that is publicly stated.
